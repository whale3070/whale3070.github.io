你认为slither找到了unstoppable的漏洞了吗？

Slither 只扫出了这个合约里的「通用风险点/代码不规范」，但完全没命中unstoppable这个场景的「核心致命漏洞」 —— 这也恰恰是你写自定义Slither插件的价值所在（官方Slither对「业务逻辑型漏洞」的检测能力极弱，而这正是DvD系列的核心考点）。

先拆解清楚：

第一步：先明确unstoppable的核心漏洞是什么（DvD里的关键考点）

unstoppable的核心是「非预期的状态篡改导致合约DOS（拒绝服务）」，核心逻辑：

  1. 金库合约（UnstoppableVault）用totalAssets()计算资产总量，其中依赖balanceOf(address(this))（金库自己的代币余额）和totalSupply（金库份额总量）的比值；
  2. 攻击者可以直接给金库转账少量代币（不走金库的deposit函数，而是直接调用ERC20的transfer）；
  3. 这会导致balanceOf(address(this)) > totalSupply，触发金库flashLoan函数里的assert断言失败，整个金库的闪电贷功能彻底卡死（DOS），任何人都无法再调用闪电贷——这才是这个场景的「核心漏洞」。

第二步：看Slither的告警，为什么没命中核心漏洞？

Slither扫出的所有告警，都是「通用型代码风险」，和上述核心漏洞无关：

| Slither告警类型 | 和核心漏洞的关系 | 通俗解释（为什么不是核心）|
| arbitrary from in transferFrom | 无关 | 只是transferFrom的from参数可控，和DOS漏洞没关系 |
| controlled delegatecall | 无关 | execute函数用了可控的delegatecall，但这个函数不是漏洞触发点 |
| divide before multiply | 无关 | 数学库的运算顺序问题，不影响核心逻辑 |
| dangerous strict equality | 无关 | ERC4626里的supply == 0严格相等，是规范问题，不是漏洞 |
| unused return | 无关 | 忽略approve/flashLoan返回值，是代码不规范，不导致DOS |

简单说：Slither的检测逻辑是「匹配已知的代码特征（比如transferFrom任意from、忽略返回值）」，但unstoppable的核心漏洞是「业务逻辑设计缺陷」—— 没有对应的“代码特征”（既没有奇怪的调用，也没有不规范的语法），而是「合约对“资产总量计算”的逻辑假设被打破」，所以官方Slither完全识别不出来。

第三步：这恰恰是你写自定义插件的绝佳场景！

你可以基于这个点，写一个「Unstoppable漏洞专属检测器」，核心检测逻辑可以设计成：

  1. 找到合约中依赖balanceOf(address(this))计算资产总量的函数（比如totalAssets()）；
  2. 检查该函数是否「只认通过deposit/mint函数增加的资产，而未排除直接转账的资产」；
  3. 检查flashLoan等核心函数是否依赖totalAssets()的结果做assert/require判断；
  4. 如果满足以上条件，直接标红告警：「资产总量计算未排除外部直接转账，存在DOS风险（Unstoppable漏洞）」。

这个插件写出来后，对比官方Slither的结果——官方Slither没发现核心漏洞，而你的插件能精准命中，这会是你求职时的「高光案例」：

  • 能证明你懂「工具的局限性」；
  • 能证明你懂「业务逻辑型漏洞的检测思路」；
  • 能证明你能「定制工具解决实战问题」—— 这正是初级审计岗最看重的能力。

总结

Slither只是做了「基础的代码体检」，没找到unstoppable的核心漏洞；但这对你来说是好事—— 官方工具的短板，恰恰是你展现自己能力的机会。你把这个场景的插件写出来，就能完美体现「你不仅会用工具，还能弥补工具的不足」，这在面试中会直接拉开和其他候选人的差距。

如果要写这个插件，核心要抓的点是：检测「合约是否将外部直接转账的代币计入资产总量，且基于此做刚性断言」 —— 你之前写0day POC时抓「触发条件」的思路，完全可以复用在这里～

我们现在来看下slither的源代码，哪些是放插件的地方？

Slither的「插件/自定义检测器」核心存放和加载路径非常清晰，结合你的插件开发经验（sqlmap/Xray），我会把「官方检测器位置」「自定义插件位置」「加载逻辑」都讲透，你能直接复刻写插件～

一、先明确：Slither里的「插件」本质是「自定义检测器（Detector）」

Slither没有单独的「plugin」目录，所有漏洞检测逻辑都以「Detector（检测器）」的形式存在——你写的自定义插件，本质就是「继承Slither抽象检测器类的Python脚本」，和你给Xray写POC、给sqlmap写tamper插件的思路完全一致。

二、核心路径拆解（从源码结构到自定义插件）

先看Slither源码的核心目录结构（聚焦检测器相关）：

1. 官方检测器的位置（参考学习用）

你之前扫unstoppable时看到的告警，对应的检测器都在detectors/下：

  • arbitrary from in transferFrom → detectors/erc20/arbitrary_from.py
  • controlled delegatecall → detectors/delegatecall/controlled_delegatecall.py
  • unused return → detectors/unused_return/unused_return.py

这些官方检测器是你写插件的「最佳参考模板」—— 比如打开arbitrary_from.py，能直接抄它的「类定义、检测逻辑、告警输出」格式。

2. 自定义插件的存放位置（二选一，推荐第二种）

方式1：直接丢到detectors/下（简单，但和官方检测器混在一起）

把你写的unstoppable_detector.py直接放到slither/detectors/目录下，Slither启动时会自动扫描该目录下所有继承AbstractDetector的类，无需改任何配置。

方式2：创建detectors/custom/目录（推荐，隔离官方和自定义）

① 在slither/detectors/下新建custom文件夹；
② 把你的自定义插件（比如unstoppable_dos.py）放到custom/下；
③ （可选）在custom/下建__init__.py（空文件即可，标记为Python包）；
④ Slither会自动递归加载detectors/下所有子目录的检测器，无需额外配置。

👉 对你来说，方式2更清晰——后续你写18个DvD插件时，可在custom/下按漏洞分类（比如custom/reentrancy/、custom/unstoppable/），和你之前给Xray分POC目录的思路一致。

五、关键补充（适配你的开发习惯）

总结

Slither的插件体系对你来说完全是「轻车熟路」—— 核心就是「继承抽象类+实现detect逻辑+放到detectors目录下」，和你给sqlmap写插件、给Xray写POC的模式几乎一致。你只需聚焦「把DvD漏洞的业务逻辑转化为代码检测规则」，插件的加载/运行Slither都帮你做好了，不用操心底层细节～

    地址：crytic/slither
