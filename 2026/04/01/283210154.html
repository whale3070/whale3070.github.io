5. 自动先落盘当前章节内容

修改内容之后，它好像不知道怎样的生效方式，我试了好几遍，不知道是要先撤回本章再修改才会更新，还是是有个时间差才会更新（已经了解了得点击切换到别的章节才会更新）|目前又发现似乎发布新章节也是一样，必须得要切出去才能正常显示，否则发布的就是空章节。你告诉我，修改内容，已发布的内容什么时候生效？什么时候读者端能看到？

你现在遇到的现象，和当前实现一致，不是你操作错了。

现在的生效规则（按当前代码）

  • 你在编辑器里打字后，2 秒防抖保存的是“写作草稿上下文”（/api/v1/save-draft），不是章节结构发布源。
  • 读者端展示读取的是章节结构里的 chapterHtml（/api/v1/library/articles 从 .data/structure 读），再按 publishedChapterIds 过滤已发布章节。
  • 所以：
    • 只改内容但不触发章节结构写入时，读者端会看到旧内容（甚至空内容）。
    • 你切到别的章节时会触发“把当前章节内容写回结构”，所以你观察到“切出去才生效”。

回答你两个关键问题

  • 已发布内容什么时候生效？
    当“当前章节内容已经写入结构文件”后就生效。当前最稳定触发点是：切换章节（或你刚用的 Markdown “发布本章节”流程）。

  • 读者端什么时候能看到？
    在作品是公开、且该章节已在 publishedChapterIds 内、并且结构写入完成后，读者端刷新基本可立即看到（接口是 no-store，不是长缓存导致的延迟）。

为什么新章节会发布成空章

新章节创建后，如果你直接点发布，但“当前编辑内容还没先写回章节结构”，发布时拿到的还是旧的/空的 chapterHtml，所以读者端就显示空章。

如果你要，我可以直接再改一版“强一致”逻辑：
点击“发布本章 / 一键发布全部”前，自动先落盘当前章节内容，再执行发布，这样就不需要手动切章节了。

6. 章节内容错乱

章节内容有时候显示的是最新版的，有时候显示的又是旧版本的，有时候显示的有图片，有时候显示的又没有图片，但是一刷新又会恢复到最新版本

你这个现象我确认了，根因有两层：

  • 读者端不是实时订阅：页面只在进入/刷新时拉一次内容，所以作者改完后，读者通常要刷新才会看到。
  • 章节有多份 HTML 字段：之前编辑时只更新了 chapterHtml，而读者端有时会优先读 chapterHtmlDesktop / chapterHtmlMobile，就会出现“有时旧版、有时无图”。

我刚刚已经修了第二层（核心一致性问题）：

  • 在作者端保存章节时，统一同步更新：
    • chapterHtml
    • chapterHtmlDesktop
    • chapterHtmlMobile

这样同一章节不会再出现“不同入口读到不同版本”的情况，图片也不会因为读到旧字段而忽隐忽现。

现在的生效口径（你可以按这个理解）

  • 作者点击发布相关操作前，系统会先落盘当前章节内容（我们前面已加）。
  • 读者端看到更新的时间点：下次请求文章详情时（最直接就是刷新页面）。
  • 不刷新时，读者端仍可能停留在当前页面内存快照，这是预期行为（不是没保存）。

如果你希望“读者不刷新也自动看到最新”，我可以再给你加一个轻量自动刷新（比如每 5-10 秒静默拉新，并尽量保持当前阅读章节位置不跳）。

9. AI排版bug

你这个“AI自动排版有问题”的判断是对的，我查完调用链后，先给你结论（本次未改代码）：

主要问题（高优先级）

  • 自动排版是“纯文本重写”策略，不是 HTML 保留策略
    在 deepseek-publish-format 里，先把章节 chapterHtml 转成纯文本，再让模型排版，再重新生成 <p>。
    这会天然丢失图片、表格、复杂排版（例如 img/table/custom html）。
    关键位置：apps/web/src/lib/server/deepseek-publish-format.ts 里的 htmlToPlainText() + plainTextToHtml() 流程。

  • “一键发布全部章节”必走自动排版
    publish_all_chapters 在 novel-publish 路由里总是调用 autoFormatChaptersForPublish()，没有跳过选项。
    所以你一键发布后，之前保留好的富文本也可能被重新“文本化”。
    位置：apps/web/src/app/api/v1/novel-publish/route.ts。

  • 按章节发布只有部分路径支持跳过自动排版
    你之前用 Markdown 弹窗“发布本章节”是带 skipAutoFormat=true 的，这条路径可以保留渲染结果；
    但其他发布入口（尤其一键发布）仍可能触发自动排版重写。
    位置同上 + apps/web/src/components/novel-editor.tsx 的调用差异。

次要问题（中优先级）

  • DeepSeek 调用缺少超时和可见诊断
    formatByDeepSeek() 没有超时控制，慢网时可能卡很久。
  • 长章节被截断到 12000 字符
    MAX_MODEL_INPUT_CHARS = 12000，长章节会被裁切后再排版，可能导致“后半章没按预期处理”。

你现在看到“有时正常、有时异常”的根本原因

  • 不是同一条发布路径：不同按钮触发的发布策略不一样（有的保留 HTML，有的会文本化重排）。
  • 一旦走到自动排版，就可能把图片/复杂格式丢掉，看起来像“随机失效”。

如果你同意，我下一步可以给你一版“排版策略分层”方案（先给方案，不直接改）：

  • 保留富文本发布（默认）

目标

这个网站的前后端构架是什么？

整体上这是一个 Next.js 全栈一体化（BFF） 架构，前后端都在同一个 apps/web 项目里。

  • 前端层

    • Next.js 16 + React 19（App Router）
    • 页面和交互主要在 app/*、components/*
    • UI 侧用了 Tailwind、Radix、lucide-react
    • 编辑器是 TipTap + marked（Markdown/HTML 双向处理）

  • 后端层（同仓库）

    • 使用 Next 的 Route Handlers（app/api/v1/**/route.ts）当 API 层
    • 运行时是 Node（你代码里很多 export const runtime = "nodejs"）
    • 相当于“前端 + API 网关/业务层”合一，不是单独的 Spring/Nest 服务

  • 数据存储层

    • 目前不是传统数据库（MySQL/Postgres），而是本地文件持久化
    • 大量业务数据写在 .data/* 下的 JSON 文件（如 publish、draft、structure、translations）
    • 属于轻量存储方案，适合快速迭代原型/中小规模

  • AI 能力层

    • 通过服务端 API 调用 DeepSeek（chat/completions）
    • 用于：自动切章、发布元信息生成、翻译、英文净化等

  • 认证与身份

    • 钱包身份（MetaMask / Web3）为主
    • 前端 wagmi + viem，后端通过 x-wallet-address 做鉴权与隔离

  • 仓库组织

    • 看起来是 monorepo 结构（apps/, services/, shared/）
    • 当前线上核心是 apps/web 这套 Next 全栈应用

如果你愿意，我可以再给你画一个“请求链路图”（例如：workspace 点翻译 -> API -> DeepSeek -> .data/translations -> library 分语言读取）。

1) 先定义“活跃”的口径（最重要）

  • DAU（钱包）：当天有任意行为的钱包数
  • WAU/MAU（钱包）：近 7/30 天有行为的钱包数
  • 活跃行为构成：创作、发布、翻译、读者解锁、阅读访问 等各自钱包数

你可以先用一个统一口径：
“调用任一业务 API 且带有效 x-wallet-address 就记活跃”。

2) 事件埋点位置（后端统一记，不靠前端）

在这些 API 入口记一条事件（钱包、动作、时间）：

  • 创作编辑相关：/api/v1/save-draft、/api/v1/update-structure
  • 发布相关：/api/v1/novel-publish*
  • 翻译相关：/api/v1/novel-translation/*
  • 读者侧付费/解锁：/api/v1/library/articles 的 POST
  • 你认为关键的其它入口

事件格式建议：

  • wallet（lowercase）
  • eventType（如 save_draft, translate, publish）
